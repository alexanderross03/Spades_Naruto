3 Architecture Approaches

Pick the one that fits best — or let me know your thoughts

A

Next.js + Pusher + Vercel KV ★ Recommended

Frontend and API routes in Next.js on Vercel. Pusher handles real-time (WebSockets). Vercel KV (free Redis) holds game state server-side so reconnects work perfectly and cheating isn't possible.

Pros

  • Server-authoritative (secure)
  • Reconnect-safe — refresh and pick back up
  • Vercel KV is 1-click, free, no schema
  • Everything in one repo, deploys to Vercel

Cons

  • Needs a Vercel KV instance (but it's one click)
  • Slightly more moving parts
B

Next.js + Pusher (no storage)

Same as A but game state lives only in the browsers. One player's tab acts as the "host" and broadcasts state to others. No KV needed.

Pros

  • Truly no backend storage at all
  • Simpler setup

Cons

  • If the host player refreshes mid-game, state is lost
  • Other players can't reconnect cleanly
  • Easier to cheat (client holds full deck)
C

Next.js + Supabase Realtime

Uses Supabase's free tier for both real-time messaging AND storage. One service instead of two, generous free tier.

Pros

  • One service does everything
  • Free tier is very generous

Cons

  • Supabase account + project setup required
  • More overhead than needed for a small game
  • If you said "no database" — Supabase is a Postgres DB

Click to select, then let me know in the terminal. I recommend A — Vercel KV is just a key-value store (JSON blobs), not a real "database" in the scary sense. No schema, no tables, one click to create.
